# 存档与同步策略

> 更新时间：2026-03-07
> 状态：已完成（v1.0）

## 1. 目标与范围

### 1.1 目标
- 定义 IdleWoool 在 Web 场景下的存档一致性策略，保证“挂机结算、资源变更、装备变更、进度推进”可恢复、可追溯、可幂等。
- 明确“客户端本地缓存”和“服务端权威存档”的边界，避免多端或弱网场景下的数据冲突。
- 为 `34-api-contract.md`、`33-database-sqlite3-schema.md`、`41-testing-strategy.md` 提供可执行同步口径。

### 1.2 范围
- In Scope：
  - 单账号在多浏览器标签页/多设备登录下的最终一致性策略。
  - 在线增量保存、关键事件强制保存、离线收益领取前后校验。
  - 冲突检测（版本号）与冲突处理（服务端合并优先级）。
  - 异常场景（断网、超时、重复提交、时钟漂移）处理。
- Out of Scope：
  - 实时联机对战同步（非本项目 MVP 范围）。
  - 跨大区迁服与跨服交易。

## 2. 存档边界与权威源

### 2.1 数据分层
- 服务端权威层（必须落库）：
  - 玩家核心进度：关卡、经验、等级、职业/技能等级。
  - 经济资产：金币、钻石、材料、离线奖励可领取状态。
  - 装备状态：背包、穿戴位、强化等级、词条随机种子。
  - 审计字段：`version`、`updated_at`、`last_op_id`。
- 客户端缓存层（可丢失）：
  - UI 展示状态（筛选、排序、面板开合）。
  - 临时战斗表现缓存（动画播放进度、本地草稿）。

### 2.2 权威原则
- 服务端为唯一真相源（Source of Truth）。
- 客户端不得直接提交“最终数值结果”，仅提交“操作意图”或“领取请求”。
- 所有写操作必须携带 `X-Request-Id`（见 `34-api-contract.md`），用于幂等去重。

## 3. 保存策略

### 3.1 触发时机
- 周期增量保存：每 30 秒触发一次轻量保存（仅脏字段）。
- 关键事件强制保存：
  - 装备穿戴/强化/分解
  - 技能升级
  - 关卡结算
  - 离线奖励领取
- 会话结束保存：用户主动退出、页面 `visibilitychange` 进入隐藏态时尝试提交一次。

### 3.2 保存粒度
- 轻量保存：资源余额、进度快照、客户端 `base_version`。
- 全量保存：在关键事件后按“玩家聚合根”写入相关表（玩家、背包、装备、技能、审计日志）。

### 3.3 性能约束
- 写接口 P95 < 200ms（本地 SQLite3 + 单实例服务目标）。
- 单玩家连续写入限流：最多 5 次/秒，超限返回可重试错误码。

## 4. 同步策略

### 4.1 版本模型
- 每个玩家存档维护单调递增 `version`。
- 客户端请求携带 `base_version`，服务端处理成功后返回 `new_version`。
- 若 `base_version` 落后，返回 `CONFLICT_VERSION_MISMATCH` 并附带最新快照摘要。

### 4.2 同步流程（在线）
1. 客户端拉取 `GET /v1/player/snapshot`（初始或冲突后）。
2. 客户端提交写请求（附 `X-Request-Id` + `base_version`）。
3. 服务端事务内校验版本、执行规则、写库、递增版本。
4. 返回 `new_version` 与最小必要回包字段。
5. 客户端更新本地快照并清理脏标记。

### 4.3 同步流程（离线恢复）
1. 客户端恢复会话后先拉快照，不上传本地离线期间“推导结果”。
2. 服务端基于 `last_active_at` 与当前时间计算离线窗口（参见 `13-system-offline-rewards.md`）。
3. 玩家发起领取时，通过幂等键保证“最多一次有效发放”。
4. 成功后返回新版本与领取结果明细。

## 5. 冲突处理与合并规则

### 5.1 冲突判定
- 判定条件：`base_version != current_version`。
- 典型场景：
  - 多标签页同时操作装备。
  - 移动端和桌面端同时领取离线奖励。

### 5.2 合并原则
- 资源类字段：以服务端已提交版本为基线重新计算，不做客户端覆盖。
- 背包/装备变更：按操作日志重放，冲突操作返回失败并要求刷新。
- UI 偏好类字段：允许客户端最后写入覆盖（不影响经济与战斗平衡）。

### 5.3 自动修复策略（无需人工）
- 客户端遇到 `CONFLICT_VERSION_MISMATCH`：自动拉取最新快照并重放“可重放意图”一次。
- 若重放仍失败：丢弃本地意图并提示“数据已刷新”。
- 重放次数上限 1 次，防止死循环。

## 6. 可靠性与安全约束

### 6.1 幂等与防重
- 所有写接口强制 `X-Request-Id`。
- 服务端在 `idempotency_records` 保留 24h 请求摘要；重复请求直接回放首次结果。

### 6.2 事务一致性
- 单次关键写操作必须在同一事务提交：
  - 资源扣减 + 奖励发放 + 审计日志
  - 装备强化结果 + 材料消耗 + 背包变更

### 6.3 风险防护
- 客户端时间不作为经济结算依据。
- 对高风险写操作记录审计日志（玩家ID、请求ID、前后版本、变更摘要）。
- 单账号异地高频写入触发风控标记（仅告警，不阻塞 MVP）。

## 7. 异常场景与降级

### 7.1 网络中断
- 客户端将写意图入本地队列（内存 + 可选 localStorage）。
- 恢复后按顺序重放；每条意图保持原 `X-Request-Id`。

### 7.2 服务端超时
- 客户端仅允许“查询后重试”模式：先拉快照确认版本，再决定是否补发请求。
- 禁止无查询盲重试，避免重复结算。

### 7.3 存档损坏（极低概率）
- 使用最近一次成功快照 + 审计日志增量重建。
- 若重建失败，回滚到最近可用版本并发放补偿邮件（后续实现项，文档先占位）。

## 8. 配置与实现约束

### 8.1 配置项
- `save.interval_sec = 30`
- `sync.retry.max_replay = 1`
- `idempotency.ttl_hours = 24`
- `conflict.auto_pull = true`

### 8.2 代码落地建议
- 前端：
  - `saveQueue`（写意图队列）
  - `snapshotStore`（当前权威快照）
  - `syncService`（重放与冲突处理）
- 后端：
  - `SaveService`（保存入口）
  - `SyncService`（快照拉取/版本校验）
  - `IdempotencyService`（防重与结果回放）

## 9. 验收标准（DoD）
- 能通过“多标签页并发写入”测试，冲突率可观测、冲突后可自动恢复。
- 写接口重复请求不产生双倍奖励/双倍扣费。
- 离线奖励领取在超时重试场景下仍保持幂等。
- 关键链路日志可追溯到 `player_id + request_id + version`。
- 与 `34-api-contract.md`、`33-database-sqlite3-schema.md` 字段口径一致。
