# 前端架构（Next.js）

> 状态：已完成（v1.0）
> 更新时间：2026-03-07
> 关联阶段：D3.1 架构蓝图
> 关联文档：`30-architecture-overview.md`、`32-backend-fastapi-architecture.md`、`34-api-contract.md`、`35-save-and-sync-strategy.md`

## 1. 目标与范围

### 1.1 目标
- 建立可长期演进的 Next.js 前端分层架构，保障“挂机 + 养成”高频状态更新场景下的可维护性与性能稳定性。
- 统一页面、状态、接口、埋点、异常处理口径，降低多人协作时的实现分歧。
- 为后续 API 契约、联调与测试提供可执行前端基线。

### 1.2 范围（MVP）
- 使用 Next.js App Router 组织页面与路由。
- 使用 BFF（Next.js Route Handlers）承接前端会话、聚合接口与降级兜底。
- 使用 Zustand + React Query 管理本地状态与服务端状态。
- 覆盖核心页面：登录/角色、主城、战斗挂机、背包装备、离线收益结算。

### 1.3 非范围（本阶段不拍板）
- 不引入微前端拆分。
- 不引入复杂 SSR SEO 策略（游戏内页以交互优先）。
- 不绑定具体 UI 组件库（只规定设计令牌与交互约束）。

## 2. 分层架构

### 2.1 架构总览
- 表现层（UI）：页面与组件，仅负责渲染与交互事件上报。
- 应用层（Application）：组合用例（登录、领取离线收益、装备切换），负责编排流程。
- 领域层（Domain）：角色、战斗、装备、离线收益等领域模型与纯函数规则。
- 基础设施层（Infra）：HTTP 客户端、缓存、埋点、错误上报、配置读取。

### 2.2 目录建议
```text
src/
  app/
    (auth)/
    (game)/
    api/
  features/
    combat/
    profession/
    equipment/
    offline/
  entities/
    player/
    item/
    battle/
  shared/
    ui/
    lib/
    config/
    telemetry/
```

### 2.3 依赖方向约束
- `app` 可依赖 `features/entities/shared`。
- `features` 可依赖 `entities/shared`，不可跨 feature 直接耦合。
- `entities` 仅依赖 `shared`。
- `shared` 不依赖业务模块。

## 3. 渲染与路由策略

### 3.1 渲染策略
- Shell 页面采用 SSR/Streaming，保证首屏可用。
- 游戏核心交互区域采用 CSR，保障高频刷新与交互性能。
- 高频数值（DPS、资源跳字）在客户端增量计算，服务端定期校准。

### 3.2 路由与布局
- `app/(auth)`：登录、角色创建与进入游戏。
- `app/(game)`：主城与功能页，使用统一 `GameLayout`（顶部资源栏 + 左导航 + 主内容区）。
- `app/api/*`：BFF 接口层，处理会话透传、字段裁剪与错误归一。

### 3.3 导航与守卫
- 未登录访问 `(game)` 自动跳转 `(auth)`。
- 角色状态缺失时引导创建角色流程。
- 关键操作（装备分解、消耗材料）采用二次确认弹层。

## 4. 状态管理与数据流

### 4.1 状态分类
- 服务端状态：角色快照、背包、战斗结算、离线收益（React Query）。
- 客户端状态：UI 开关、筛选条件、临时拖拽状态（Zustand）。
- 持久化状态：低风险偏好配置（音量、显示模式），存储于 LocalStorage。

### 4.2 Query Key 规范
- `player.profile`
- `player.resources`
- `battle.snapshot`
- `equipment.inventory`
- `offline.pendingReward`

### 4.3 一致性策略
- 关键变更（装备穿戴、领取离线收益）采用“乐观更新 + 回滚”策略。
- 结算类结果以服务端返回为准，前端仅做展示层补间。
- 跨页共享数据统一通过 Query Cache，不通过全局事件总线绕过。

## 5. BFF 与接口调用约束

### 5.1 BFF 职责
- 统一鉴权头与会话令牌透传。
- 聚合多后端接口减少首屏瀑布请求。
- 对外输出稳定前端 DTO，屏蔽后端内部字段演化。

### 5.2 错误归一
- 统一错误结构：`{ code, message, retryable, traceId }`。
- `retryable=true` 时前端展示“可重试”交互；否则展示明确失败原因。
- 统一携带 `traceId` 便于联调追踪。

### 5.3 超时与重试
- 普通查询超时：3s，最多重试 1 次（指数退避）。
- 关键结算接口（领取奖励）默认不自动重试，避免重复操作风险。

## 6. 性能与体验基线

### 6.1 性能目标（MVP）
- 首屏可交互时间（TTI）<= 3s（常规网络）。
- 主循环界面稳定 30 FPS 以上（中端设备）。
- 单次路由切换中位耗时 <= 300ms。

### 6.2 优化策略
- 大组件按路由与功能分块懒加载。
- 图标与静态资源统一走 CDN 路径。
- 高频列表（背包）使用虚拟滚动，避免长列表重渲染。

## 7. 可观测性与质量保障

### 7.1 前端埋点
- 核心埋点：登录成功、进入战斗、领取离线收益、装备强化、错误弹窗触发。
- 埋点字段与事件命名在 `42-observability-and-logs.md` 统一收口。

### 7.2 错误监控
- 运行时错误、接口错误、资源加载失败统一上报。
- 采样策略：同类错误 5 分钟内聚合上报，避免噪音。

### 7.3 测试要求
- 关键流程（登录、战斗、领取离线收益）必须具备 E2E 用例。
- 领域纯函数（数值显示换算、资源格式化）必须具备单测。

## 8. 安全与防作弊前端边界

- 前端不作为数值真值来源，所有关键结算由服务端裁决。
- 前端时间仅用于展示，不参与离线收益最终计算。
- 本地缓存仅存非敏感偏好，不存放可篡改的关键状态快照。
