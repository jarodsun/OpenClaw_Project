# 41 测试策略与质量门禁

更新时间：2026-03-07

## 1. 目标与范围

- 建立覆盖前端（Next.js）、后端（FastAPI）、数据库（SQLite3）与核心玩法链路的分层测试体系。
- 将文档阶段已确定的规则（战斗、掉落、离线收益、存档同步、幂等）转化为可执行测试用例。
- 为后续实现阶段提供统一质量门禁，确保“可运行”与“可发布”分层验收。

## 2. 测试分层

### 2.1 单元测试（Unit）
- 目标：验证纯逻辑模块正确性，覆盖核心公式与边界条件。
- 覆盖对象：
  - 战斗与成长计算（见 `10-system-combat-and-progression.md`）
  - 职业/技能规则（见 `11-system-professions-and-skills.md`）
  - 装备词条与掉落权重（见 `12-system-equipment-and-loot.md`）
  - 离线收益计算（见 `13-system-offline-rewards.md`）
- 通过标准：核心公式模块语句覆盖率 >= 90%，分支覆盖率 >= 80%。

### 2.2 集成测试（Integration）
- 目标：验证服务层、数据库层与配置层的协同一致性。
- 覆盖对象：
  - FastAPI 路由 -> Service -> Repository -> SQLite 事务
  - 幂等写接口（`X-Request-Id`）重复请求处理
  - 配置热更加载 + 版本兼容校验
- 通过标准：核心接口集成用例通过率 100%，无阻断级数据一致性缺陷。

### 2.3 端到端测试（E2E）
- 目标：验证玩家主路径体验与关键业务闭环。
- 覆盖对象（最小闭环）：
  - 新账号初始化 -> 战斗推进 -> 掉落装备 -> 提升战力
  - 离线返回 -> 领取收益 -> 存档保存 -> 刷新恢复
  - 配置版本升级后登录与存档兼容
- 通过标准：主路径成功率 100%，阻断级缺陷为 0。

### 2.4 非功能测试
- 性能：核心读接口 P95 < 200ms，关键写接口 P95 < 300ms。
- 稳定性：关键回归用例连续执行 50 轮无随机失败。
- 安全性：关键写接口具备鉴权、幂等、防重放最小校验。

## 3. 用例设计方法

### 3.1 规则映射矩阵
- 每个系统文档至少映射“正常路径 + 边界路径 + 异常路径”三类测试。
- 每条核心规则需有唯一测试编号，便于缺陷追踪与回归选择。

### 3.2 边界与异常优先级
- P0：经济系统与成长一致性（货币、经验、战力、装备产出）。
- P1：同步冲突、幂等重试、离线补偿。
- P2：显示层与非阻断体验问题。

### 3.3 数据管理
- 使用固定种子测试数据，确保结果可复现。
- 测试数据库与开发数据库隔离，禁止共用同一 SQLite 文件。
- 每次集成测试后自动清理临时数据，避免污染。

## 4. 质量门禁（DoD）

### 4.1 提测门禁（进入联调）
- 单元测试全部通过。
- 核心接口集成测试全部通过。
- 新增/修改功能必须附带对应测试。

### 4.2 发布门禁（进入灰度/正式发布）
- E2E 主路径全部通过。
- P0/P1 缺陷清零，P2 有明确修复计划与风险评估。
- 性能指标达标，且无新增高危安全问题。

## 5. 自动化执行策略

- 本地：开发者提交前执行最小回归（单元 + 关键集成）。
- CI：按 `frontend` 与 `backend` 分流水线并行执行。
- 夜间回归：每日定时执行全量回归 + E2E + 性能基线检测。
- 失败处理：同一用例连续失败 2 次自动升级为阻断告警。

## 6. 缺陷分级与响应

- S0（阻断）：导致数据错乱/存档损坏/核心流程不可用，立即停止发布。
- S1（严重）：核心功能可用但结果不可信，24h 内修复。
- S2（一般）：存在体验退化或局部异常，纳入迭代修复。
- S3（轻微）：文案或非关键视觉问题，按计划排期。
