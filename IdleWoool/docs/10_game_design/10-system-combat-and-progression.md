# 系统设计：战斗与成长

> 版本：v1.0
> 状态：已完成（文档）
> 适用阶段：D2.2 系统与数值

## 1. 文档目标

将 `02-game-design-core-loop.md` 的循环定义落地为可实现的战斗与成长系统规则，明确计算顺序、成长曲线、配置入口与验收标准，作为后端结算实现与前端展示逻辑的统一依据。

## 2. 设计边界

- 本文仅覆盖：战斗结算、等级成长、战力计算、区域推进。
- 本文不覆盖：职业技能细节（见 `11-system-professions-and-skills.md`）、装备词条细节（见 `12-system-equipment-and-loot.md`）、离线结算细节（见 `13-system-offline-rewards.md`）。
- 所有数值均“配置优先”，文档默认值仅作为 MVP 启动参数。

## 3. 战斗模型（MVP）

### 3.1 Tick 与回合

- 结算 Tick：`1s`。
- 每个 Tick 执行顺序：
  1) 读取角色与怪物当前状态；
  2) 执行技能判定（冷却与资源）；
  3) 执行普攻判定；
  4) 计算伤害并扣减生命；
  5) 处理死亡与掉落；
  6) 写入本 Tick 战斗日志。

### 3.2 伤害计算口径

基础伤害公式（MVP）：

`damage = max(1, atk * skill_ratio * rand(0.95, 1.05) - def * def_ratio)`

- `atk`：攻击方当前攻击力（含装备、等级、被动加成）。
- `skill_ratio`：技能倍率；普攻固定为 `1.0`。
- `def_ratio`：防御折算系数，默认 `0.6`。
- 暴击：`crit_rate` 概率触发，伤害乘 `crit_multiplier`（默认 `1.5`）。

### 3.3 战斗结果判定

- 怪物 HP <= 0 判定击杀，立即触发收益结算。
- 角色 HP <= 0 判定本次战斗失败，进入 `3s` 冷却后自动重试当前关卡。
- Boss 战失败不扣资源，仅损失时间收益。

## 4. 成长系统

### 4.1 等级成长

- 等级上限（MVP）：`100`。
- 升级经验需求：

`need_exp(level) = floor(100 * level ^ 1.35)`

- 升级收益：每级固定增加 `HP/ATK/DEF` 基础属性；系数由职业模板提供。

### 4.2 成长行为优先级

每次 `Settle` 后按如下顺序执行：

1. 经验入账并判定升级；
2. 金币入账；
3. 掉落入背包；
4. 自动穿戴更优装备；
5. 刷新战力快照；
6. 判定关卡解锁。

说明：固定顺序用于保证“同输入必得同结果”，降低联调分歧。

### 4.3 防卡顿保护（Anti-Stall）

- 若连续 `N=30` 次结算未触发“升级/换装/技能升级”，触发一次成长提示事件。
- 成长提示事件不直接发奖，仅提示推荐动作（强化/回刷低级图/切换技能策略）。
- 该机制用于缓解挂机“无反馈”疲劳，不改变经济平衡。

## 5. 战力计算

统一战力公式（展示与自动穿戴共用）：

`power = hp_weight * HP + atk_weight * ATK + def_weight * DEF + speed_weight * SPEED + skill_bonus`

默认权重：

- `hp_weight = 0.2`
- `atk_weight = 1.0`
- `def_weight = 0.6`
- `speed_weight = 0.8`

约束：

- 权重统一配置在 `balance.combat.power_weights`。
- 自动穿戴仅在 `new_power > old_power` 时触发。

## 6. 区域推进

- 关卡解锁条件：角色等级达到阈值且前置关卡通过。
- 收益单调要求：后续关卡 `exp_per_min`、`gold_per_min` 的期望值不低于前一关。
- 失败回退：允许回刷已通关关卡，不设置体力限制。

## 7. 配置与数据结构入口

依赖配置域（详见后续 `21-content-config-schema.md`）：

- `combat.global`：Tick、伤害浮动、暴击参数。
- `growth.level_curve`：升级经验曲线与等级上限。
- `growth.level_attr`：每级属性增长模板。
- `combat.power_weights`：战力计算权重。
- `stage.progression`：关卡解锁条件与收益期望参数。

## 8. 实现约束（前后端）

- 后端是战斗与成长结算唯一可信源。
- 前端只做展示与交互，不在本地持久化最终结算结果。
- 所有奖励发放接口必须幂等，重复请求不可重复发奖。
- 战斗日志保留最近 200 条（MVP），用于问题追踪与验收。

## 9. 验收标准

满足以下条件可判定“战斗与成长系统文档可用于开发”：

1. 战斗计算顺序、伤害口径、失败处理规则清晰且无冲突；
2. 升级曲线、战力公式、自动穿戴触发条件明确；
3. 关卡推进收益单调性有显式约束；
4. 可直接支撑 `34-api-contract.md` 的结算接口定义与 `41-testing-strategy.md` 的测试用例设计。
