# 总体架构设计（Architecture Overview）

> 状态：已完成（v1.0）
> 更新时间：2026-03-07

## 1. 文档目标

给出 IdleWoool 首期可落地的总体架构蓝图，统一前端、后端、数据库、配置与运维边界，确保后续 `31~35` 文档按同一口径展开。

## 2. 架构目标与约束

### 2.1 目标

1. 以“单体优先、可演进”为原则，优先保障首版交付速度。
2. 支撑核心循环稳定运行：在线战斗、离线结算、装备与成长。
3. 确保关键链路可观测、可回滚、可审计。

### 2.2 约束

- 前端：Next.js（Web 单端）
- 后端：FastAPI（单服务）
- 数据库：SQLite3（首版）
- 配置：文件化配置 + 版本号管理（对齐 `21-content-config-schema.md`）

## 3. 总体架构视图

```
[Browser / Next.js]
   | HTTPS + Session Token
   v
[FastAPI Monolith]
   |- Auth & Session
   |- Combat & Progression
   |- Loot & Equipment
   |- Offline Reward Settlement
   |- Profile/Inventory/Tasks
   |- Admin Config APIs
   |
   |- Config Loader (versioned JSON)
   |- Scheduler/Background Jobs
   v
[SQLite3]
   |- Player & Progress Tables
   |- Battle/Reward Logs
   |- Config Version Metadata
```

## 4. 分层与模块边界

### 4.1 前端层（Next.js）

- 职责：页面渲染、状态展示、交互触发、请求编排。
- 不负责：核心数值计算、离线时长计算、奖励最终裁定。
- 关键页面：挂机主界面、背包装备、技能成长、离线结算弹窗。

### 4.2 应用层（FastAPI）

- 职责：鉴权、业务编排、规则执行、幂等控制。
- 关键能力：
  - 战斗结算与推进（对齐 `10-system-combat-and-progression.md`）
  - 职业技能处理（对齐 `11-system-professions-and-skills.md`）
  - 装备掉落与强化（对齐 `12-system-equipment-and-loot.md`）
  - 离线收益发放（对齐 `13-system-offline-rewards.md`）
- 关键原则：服务端权威结算，客户端仅作为展示与操作入口。

### 4.3 数据层（SQLite3）

- 职责：持久化玩家状态、背包装备、结算日志与配置版本。
- 原则：
  - 核心写操作走事务，保证原子性。
  - 关键结算表记录 request_id / settle_id，保障幂等。
  - 高频查询字段建立必要索引，控制首版查询延迟。

### 4.4 配置与内容层

- 内容来源：`data/reference/woool_data/` + 项目配置文件。
- 配置流程：编辑 -> 校验 -> 发布版本 -> 灰度/全量生效 -> 可回滚。
- 版本策略：客户端请求携带配置版本，服务端按兼容策略处理。

## 5. 关键数据流

### 5.1 在线挂机结算

1. 前端发起结算请求（携带会话与上下文）。
2. 后端加载角色快照与配置版本。
3. 执行战斗/掉落/成长规则并写入事务。
4. 返回增量结果（经验、金币、装备变化、战力变化）。

### 5.2 离线奖励结算

1. 登录后由服务端计算离线时长（不信任客户端时间）。
2. 按离线上限与效率系数计算收益。
3. 写入幂等结算记录并更新玩家资产。
4. 返回离线汇总与关键明细用于前端展示。

### 5.3 配置热更新

1. 发布新配置版本并落库元信息。
2. 应用层按版本切换规则加载新配置。
3. 若监控异常超阈值，回滚至上一个稳定版本。

## 6. 非功能设计基线

- 性能目标：核心结算接口 P95 < 300ms（本地单机基线）。
- 稳定目标：结算幂等冲突率 < 0.1%，关键接口错误率 < 1%。
- 可观测性：必须具备请求日志、结算日志、异常日志与基础指标。
- 安全基线：参数校验、签名会话、限流、防重放与最小权限访问。

## 7. 部署拓扑（首版）

- 单机部署：`Next.js` + `FastAPI` + `SQLite3` 同机。
- 运行模式：
  - 开发：本地进程启动（对齐 `40-dev-environment-and-commands.md`）。
  - 生产首版：进程守护 + 定时备份 + 手动回滚预案。
- 备份策略：SQLite 日备份 + 发布前快照，保留最近 7 份。

## 8. 演进路线（不阻塞首版）

1. SQLite3 -> MySQL（当并发与数据量达到阈值）。
2. 单体服务 -> 轻量拆分（先拆结算与配置服务）。
3. 增加缓存层（仅在瓶颈明确后引入）。
