# 后端架构（FastAPI）

- 更新时间：2026-03-07
- 适用阶段：D3.1 架构蓝图 / D3.2 接口与数据流
- 关联文档：`30-architecture-overview.md`、`33-database-sqlite3-schema.md`、`34-api-contract.md`、`35-save-and-sync-strategy.md`

## 1. 目标与边界

### 1.1 架构目标
- 提供稳定、可回放、可审计的游戏后端能力，支撑 IdleWoool 的核心循环（战斗、成长、掉落、离线收益）。
- 在 MVP 阶段优先保证“正确性与可维护性”，性能目标以单机部署与可横向拆分为前提。
- 对外暴露一致 API 契约，避免前端与配置层对实现细节产生耦合。

### 1.2 范围边界（MVP）
- In Scope：
  - 账号会话、角色状态读取与更新
  - 战斗结算、掉落结算、离线收益结算
  - 配置版本读取与校验
  - 存档读写、基础审计日志、健康检查
- Out of Scope：
  - 分布式微服务拆分
  - 实时多人同步对战
  - 复杂反作弊策略中心（仅保留接口位）

## 2. 分层与模块设计

### 2.1 分层原则
- `API Layer`：请求解析、参数校验、鉴权、响应组装，不承载业务规则。
- `Application Layer`：用例编排（例如“领取离线收益”），协调领域服务与仓储。
- `Domain Layer`：核心规则与公式（战斗/成长/装备/离线），无框架依赖。
- `Infrastructure Layer`：SQLite 持久化、缓存、配置加载、日志与观测。

### 2.2 建议目录（MVP）
```text
backend/
  app/
    main.py
    api/
      v1/
        routers/
    application/
      services/
      dto/
    domain/
      combat/
      progression/
      equipment/
      offline/
      shared/
    infra/
      db/
      repositories/
      config/
      logging/
      cache/
    core/
      settings.py
      auth.py
      errors.py
```

### 2.3 模块职责
- `auth`：会话令牌解析、玩家上下文注入。
- `combat_service`：推进战斗 Tick、计算伤害、产出结算事件。
- `progression_service`：经验与等级更新、战力重算。
- `loot_service`：掉落池抽样、保底计数推进、装备词条生成。
- `offline_service`：按服务端会话推导离线时长并执行幂等发放。
- `save_service`：存档快照读写、版本兼容处理。

## 3. 关键流程与一致性策略

### 3.1 战斗结算流程
1. API 层校验请求（角色、关卡、幂等键）。
2. Application 层加载角色快照与配置版本。
3. Domain 层执行战斗与掉落规则，生成结算事件。
4. Infrastructure 层在单事务内提交：角色状态、背包变化、审计日志。
5. 返回统一结算响应，并记录耗时指标。

### 3.2 离线收益流程
1. 读取服务端 `last_active_at` 与当前服务端时间，计算离线时长。
2. 按 `13-system-offline-rewards.md` 公式计算收益（默认 8h / 70%）。
3. 以 `claim_id` 做幂等防重，重复领取直接返回上次结果。
4. 写入收益明细与角色增量，更新 `last_claimed_at`。

### 3.3 数据一致性
- 单玩家关键写操作使用“单事务 + 版本号”实现乐观并发控制。
- 所有结算接口必须接受幂等键（`request_id` / `claim_id`）。
- 失败重试要求“可重复执行且结果唯一”。

## 4. API 与错误模型

### 4.1 接口约束
- REST + JSON，路径版本化：`/api/v1/*`。
- 成功响应统一携带：`trace_id`、`server_time`、`config_version`。
- 高风险写接口（结算、领取）必须带幂等键。

### 4.2 错误分层
- `4xx`：参数错误、鉴权失败、状态冲突、版本不兼容。
- `5xx`：内部异常、存储不可用、配置加载失败。
- 领域错误码独立维护并在 `34-api-contract.md` 固化。

### 4.3 失败降级
- 配置中心读取失败：回退到最近可用配置版本并打告警日志。
- 非关键埋点失败：不阻塞主流程，仅记录异常。
- 审计写入失败：主流程可继续，但标记 `audit_pending=true` 供补偿任务处理。

## 5. 存储与配置协同

### 5.1 SQLite 使用边界（MVP）
- 作为单机主存储，承载角色、背包、进度、结算日志。
- 通过 WAL 模式提升读写并发；长事务必须限制在关键写路径。
- Schema 细节以 `33-database-sqlite3-schema.md` 为准。

### 5.2 配置加载策略
- 启动时加载基础配置并执行静态校验。
- 运行中按版本拉取配置快照，热更失败自动回滚上一版本。
- 规则字段与类型约束遵循 `21-content-config-schema.md`。

## 6. 可观测性与运维基线

### 6.1 日志规范
- 结构化日志（JSON），字段至少包含：`trace_id`、`player_id`、`endpoint`、`latency_ms`、`result`。
- 业务日志与审计日志分流，审计日志保留结算关键参数快照。

### 6.2 指标与告警（MVP）
- 核心指标：接口成功率、P95 延迟、结算失败率、幂等冲突率。
- 告警阈值（初版建议）：
  - 5 分钟窗口成功率 < 99%
  - 结算失败率 > 1%
  - P95 > 300ms（关键读接口）/ 600ms（结算接口）

### 6.3 健康检查
- `GET /health/liveness`：进程存活。
- `GET /health/readiness`：数据库连接、配置可用性、关键依赖状态。

## 7. 安全与反作弊最低要求

- 客户端不得提交可直接影响收益的时间戳与战斗结果终值。
- 服务端重算关键结算值，不信任客户端金额与掉落明细。
- 保留可扩展钩子：风控评分、异常行为采样、封禁策略（后续版本实现）。
