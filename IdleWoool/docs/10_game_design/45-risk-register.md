# 45 风险清单与应对（Risk Register）

## 2. 风险分级与处置策略

### 2.1 分级定义
- `R1（高）`：可能导致上线阻断、核心功能不可用、经济系统失衡或数据安全事件。
- `R2（中）`：不阻断上线，但会显著影响体验、质量成本或运营效率。
- `R3（低）`：局部影响，可通过迭代修复，不影响核心里程碑。

### 2.2 处置时效
- `R1`：24 小时内出具处置方案，必须纳入最近一个里程碑门禁。
- `R2`：72 小时内完成修复计划并排期。
- `R3`：进入常规 Backlog，按版本窗口处理。

### 2.3 状态定义
- `open`：已识别，未完成处置。
- `mitigating`：已采取缓解措施，持续观察。
- `watching`：已满足短期门禁，需持续监控。
- `closed`：风险已关闭（验证通过并归档证据）。

### 2.4 量化评分模型（复评审补充）
- 评分公式：`Risk Score = Impact(1-5) × Probability(1-5)`。
- 分档规则：
  - `15-25`：按 `R1` 管理（必须进入里程碑门禁）。
  - `8-14`：按 `R2` 管理（纳入当前迭代跟踪）。
  - `1-7`：按 `R3` 管理（常规 Backlog 管理）。
- 使用原则：当“描述等级”与“量化评分”冲突时，以更高等级执行，并在评审日志留证。

### 2.5 风险受理阈值（本轮新增）
- 受理判定：仅当风险状态达到 `watching` 或 `closed` 且满足对应证据要求，才允许从发布门禁中“受理放行”。
- `R1` 受理阈值：连续 24 小时无触发信号 + 至少 1 次演练/回归通过证据。
- `R2` 受理阈值：连续 72 小时趋势稳定（无恶化）+ 已纳入下一迭代跟踪项。
- `R3` 受理阈值：确认不影响当前里程碑，且已有明确处理窗口与 Owner。
- 不满足阈值时：状态不得标记为 `closed`，并在 `07-review-log.md` 追加“未受理原因”。

## 4. 里程碑门禁映射

### 4.1 与发布计划门禁映射
- 里程碑进入条件：所有 `R1` 风险不得处于 `open`，至少达到 `mitigating` 且有观测证据。
- Go/No-Go 判定：若存在“无缓解策略的 `R1`”或“连续超阈值未回滚”，直接 `No-Go`。
- 受理放行条件：处于 `watching` 的 `R1` 风险必须满足 `2.5` 的 `R1` 受理阈值，否则按 `open` 等价处理。

### 4.2 与测试策略门禁映射
- 提测前必须验证：`RK-002`、`RK-003`、`RK-004` 的测试用例覆盖率与通过率。
- 发布前必须验证：`RK-005`、`RK-006`、`RK-007` 的告警与回滚演练记录。

## 5. 执行流程（识别 -> 响应 -> 复盘）

### 5.1 识别
- 输入来源：评审记录、测试结果、观测告警、线上事件、用户反馈。
- 新增风险必须包含：风险描述、触发信号、等级、Owner、默认策略。

### 5.2 响应
- `R1`：立即创建应急任务，限制变更窗口，并同步到发布群与评审日志。
- `R2/R3`：纳入迭代计划，按时效跟踪执行。

### 5.3 复盘与归档
- 风险关闭条件：触发信号恢复正常 + 经过至少一轮回归验证。
- 复盘产物：事件时间线、根因、修复项、防再发项，归档至 `07-review-log.md`。
