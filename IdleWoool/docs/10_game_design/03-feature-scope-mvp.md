# MVP 范围与验收边界

> 版本：v1.0  
> 状态：已完成（文档）  
> 适用阶段：D2.1 产品与范围

## 1. 目标

在不引入重社交、强运营与复杂在线服务前提下，交付“可玩、可成长、可验证留存”的 Web 挂机版本，满足以下最小目标：

- 玩家可在 10 分钟内完成首轮“进入战斗 -> 获得收益 -> 强化 -> 再战斗”的正反馈循环。
- 支持在线挂机与离线收益结算，并可持续推进角色成长。
- 提供基础装备获取与替换、职业成长、关卡推进三条成长线。
- 可采集最小运营指标（D1 留存、平均在线时长、主线推进深度）。

## 2. 范围定义

## 2.1 In Scope（MVP 必做）

1) 账号与存档
- 游客登录（本地设备标识 + 服务端存档）。
- 单角色存档读写与基础冲突处理（以后写覆盖先写，记录更新时间）。

2) 核心玩法循环
- 自动战斗（基于 `02-game-design-core-loop.md` 的 Tick 循环）。
- 战斗结算（经验、金币、掉落）。
- 角色成长（等级、基础属性提升）。
- 离线收益（默认口径：上限 8h、效率 70%，待最终拍板）。

3) 职业与技能（最小集）
- 3 个基础职业（战士/法师/道士）可选一。
- 每职业 2 个主动技能 + 1 个被动技能。
- 技能升级仅使用单一资源（技能点），不引入多货币系统。

4) 装备与掉落（最小集）
- 部位：武器、衣服、饰品（3 槽位）。
- 品质：普通/优秀/稀有（3 档）。
- 掉落来源：普通关卡与首领关卡。
- 装备替换策略：按战力评分一键替换。

5) 关卡与怪物（最小集）
- 章节：2 章，每章 10 关，共 20 关。
- 怪物类型：普通怪、精英怪、Boss。
- 解锁规则：通关前一关解锁下一关。

6) 前端基础页面
- 主页（角色信息、当前收益、战斗状态）。
- 成长页（等级、属性、技能升级）。
- 背包页（装备查看与一键替换）。
- 离线结算弹窗（收益明细与领取）。

7) 后端基础能力
- 角色数据读写 API。
- 战斗结算与奖励发放 API。
- 离线收益结算 API。
- 基础日志（请求耗时、错误码、关键经济产出）。

## 2.2 Out of Scope（MVP 不做）

- 实时 PvP、行会/公会、组队副本。
- 拍卖行、玩家交易、邮件系统。
- 复杂活动系统（赛季、通行证、限时池）。
- 多角色系统与跨服同步。
- 完整商业化（首版仅保留可插拔点，不接入真实支付）。
- 重度反作弊（MVP 仅保留基础校验与日志审计）。

## 3. 版本边界与约束

- 客户端：Next.js Web 单端优先，移动端通过响应式兼容，不单独开发 App。
- 服务端：FastAPI + SQLite（MVP 阶段），预留未来 MySQL 迁移字段。
- 数值来源：优先复用 `data/reference/woool_data/` 与文档默认值，未定项统一进入待拍板池。
- 性能目标（MVP）：
  - 关键接口 P95 < 300ms（本地/小规模环境）。
  - 战斗与离线结算错误率 < 1%。

## 4. 里程碑验收标准

## 4.1 功能验收（必须全部满足）

- 新玩家从创建角色到通关第 3 关全流程无阻塞。
- 离线 2 小时后上线可正确领取收益，且与规则口径一致。
- 装备一键替换可在 3 秒内完成并正确刷新战力。
- 技能升级后战斗日志可见伤害变化。

## 4.2 数据验收（灰度期）

- 首日样本中，>= 70% 玩家可在 30 分钟内完成至少 1 次成长行为（升级/换装/技能升级任一）。
- D1 留存达到 25% 及以上（内部目标，未达标则优先调优数值节奏）。
- 经济数据无明显通胀（金币净产出/净消耗比值维持在 0.8~1.2）。

## 4.3 文档验收（执行侧）

- `01-product-requirements.md`、`02-game-design-core-loop.md`、本文件三者术语一致。
- 可直接衔接 `10~13` 系统文档与 `34-api-contract.md`，无关键字段缺失。

## 5. 风险与默认处理

1) 风险：离线收益口径未最终拍板。
- 默认处理：按 Q-001 默认方案 A（8h/70%）推进系统、接口与测试文档。

2) 风险：MVP 指标目标与实际用户结构不匹配。
- 默认处理：先保留目标值用于验收门槛，灰度后按真实数据回写 `20-balance-framework.md`。

3) 风险：SQLite 在并发写下出现锁竞争。
- 默认处理：MVP 保守控制写入频次并增加重试策略，D3 阶段再评估 MySQL 切换。
